# kbdiag v1.4 设计规格

## 背景

kbdiag 当前版本为 v1.3（dist/kbdiag，20 个命令），已覆盖实例状态、集群、复制、健康检查、锁、性能、日志、多节点等诊断场景。

v1.4 目标：补充 **DBA 高级工具箱**，分三个方向：

1. **运维操作**：cancel/terminate 会话与查询
2. **索引分析**：unused / dup / bloat / missing 四类索引问题
3. **统计信息与建议**：列统计深度分析 + 综合 advisor（索引/vacuum/参数/analyze 建议，`--fix` 输出可执行 SQL）

---

## 全局约束

- 纯 Shell，以 `kingbase` OS 用户运行，零侵入（不建表不装扩展）
- 所有 DB 查询通过 `ksql_q()`，禁止交互会话
- KingbaseES 特有：布尔值字符串 `true`/`false`，系统视图 `sys_` 前缀，`pg_` 函数保留前缀
- 每个新命令必须支持 `--format json`、`--quiet`、`--top N`
- 新命令归入 `[DBA]` 层（需要 DBA 背景才能解读）
- `build.sh` 打包后 `dist/kbdiag` 是唯一分发产物

## 命令详细规格

### 1. `kbdiag kill` — 会话/查询终止

**目标**：让 DBA 能精准取消查询或踢掉会话，支持按条件批量操作，强制加 `--dry-run` 防误操作。

#### 子命令与参数

```
kbdiag kill <pid>                   # cancel（软终止查询，保留连接）
kbdiag kill --terminate <pid>       # terminate（强踢连接）
kbdiag kill --long N                # cancel 运行超 N 秒的所有查询
kbdiag kill --idle-txn N            # terminate 空闲事务超 N 秒的会话
kbdiag kill --user <username>       # 按用户名过滤（可与 --long/--idle-txn 组合）
kbdiag kill --db <dbname>           # 按数据库过滤
kbdiag kill --dry-run ...           # 只列出会被终止的会话，不执行任何操作
```

**批量操作规则**：使用 `--long` 或 `--idle-txn` 时，必须先以 `--dry-run` 列出目标，工具会打印提示要求用户确认（`--force` 跳过确认，用于脚本自动化）。

#### 底层函数

| 操作 | 函数 |
|------|------|
| 取消查询（保留连接） | `sys_cancel_backend(pid)` |
| 终止连接 | `sys_terminate_backend(pid)` |

#### 输出（text 模式）

```
[OK]   Cancelled query on pid 1234 (user: appuser, duration: 312s)
[WARN] pid 5678 not found or already terminated
```

批量模式（dry-run）：

```
==> Sessions matching --long 300
  PID   USER      DB      STATE   DURATION  QUERY
  1234  appuser   test    active  312s      SELECT ...
  5678  report    test    active  480s      SELECT ...

Run without --dry-run to cancel 2 session(s).
```

#### JSON 输出

```json
[{"pid":1234,"action":"cancelled","user":"appuser","duration_s":312,"result":"ok"},
 {"pid":5678,"action":"cancelled","result":"not_found"}]
```

### 2. `kbdiag idx [sub]` — 索引健康分析

**目标**：快速定位四类索引问题，配合 `advisor --fix` 生成清理 SQL。

#### 子命令

| 子命令 | 说明 | 数据来源 |
|--------|------|----------|
| `unused` | idx_scan=0 且表行数 > 1000 的索引 | `sys_stat_user_indexes` + `sys_stat_user_tables` |
| `dup` | 相同列列表（含列顺序）的重复索引 | `sys_indexes` + `sys_attribute` |
| `bloat` | 索引膨胀率估算 > 30% | `sys_stat_user_indexes` + `sys_relation_size` / `sys_index` |
| `missing` | 高 seq_scan 大表（> 10000 行）存在 FK 列但无对应索引 | `sys_constraint` + `sys_stat_user_tables` + `sys_indexes` |

默认（无子命令）：运行全部四项。

#### 输出（text 模式，以 unused 为例）

```
==> Unused indexes (idx_scan=0, table rows > 1000)
  SCHEMA  TABLE       INDEX              SIZE    LAST_RESET
  public  orders      idx_orders_status  8192 B  2026-06-01
  public  order_items idx_oi_notes       4096 B  2026-06-01

2 unused index(es) found. Use 'advisor index --fix' to generate DROP statements.
```

#### 膨胀率估算方法

使用 `sys_stat_user_indexes.idx_blks_read` 与 `pg_relation_size` 估算，精度为"数量级"级别（不依赖 pgstattuple 扩展）：

```sql
SELECT
  indexrelname,
  pg_relation_size(indexrelid) AS index_size,
  -- 估算：(relpages - ceil(reltuples * 基准因子)) / relpages
  round(100.0 * (1 - (reltuples * 8.0) / NULLIF(relpages * 8192, 0)), 1) AS bloat_pct
FROM sys_stat_user_indexes
JOIN sys_class ON sys_class.oid = indexrelid
WHERE relpages > 0
ORDER BY bloat_pct DESC;
```

### 3. `kbdiag colstat <schema.table>` — 列统计信息深度分析

**目标**：当查询计划器选了差路径时，DBA 查看列统计是否过期或分布异常。

#### 调用方式

```
kbdiag colstat public.orders
kbdiag colstat public.orders --col status   # 只看指定列
```

#### 数据来源

- `sys_statistic`（统计信息原始数据）
- `sys_stats`（sys_statistic 的友好视图）
- `sys_stat_user_tables`（最后 analyze 时间、行数变化）

#### 输出（text 模式）

```
==> Column statistics: public.orders
Last ANALYZE: 2026-06-18 14:32:11  (3 days ago)
Rows at last ANALYZE: 1,250,000  |  Current estimate: 1,380,000  |  Drift: +10.4%

  COLUMN        TYPE     N_DISTINCT  NULL%  CORRELATION  MCV_TOP3
  id            int8     -1          0%     1.00         —
  status        varchar  6           0%     0.12         pending(42%) shipped(31%) done(18%)
  customer_id   int4     -0.92       0%     0.05         —
  created_at    timestamp -1         0%     0.98         —
  amount        numeric  -0.88       2.1%   0.15         —

[WARN] status.correlation=0.12 — poor correlation, range scans may be slow
[WARN] Drift +10.4% since last ANALYZE — consider running ANALYZE public.orders
```

#### 关键指标说明

| 指标 | 含义 | 风险信号 |
|------|------|----------|
| `n_distinct` | 负数=占比（-0.92 = 92% 唯一值），正数=绝对估算值 | 严重偏差导致行数估算错 |
| `correlation` | 列值与物理存储顺序的相关性，[-1, 1] | 接近 0 时 Index Scan 可能反而慢 |
| MCV top-3 | 最常见值及频率 | 极度倾斜时需 partial index 或 extended stats |
| Drift | 自上次 ANALYZE 以来的行数变化比例 | > 10% 时统计信息可能过期 |

#### JSON 输出

```json
{
  "table": "public.orders",
  "last_analyze": "2026-06-18T14:32:11",
  "row_drift_pct": 10.4,
  "columns": [
    {"name":"status","n_distinct":6,"null_pct":0,"correlation":0.12,"mcv":[{"value":"pending","freq":0.42}]},
    ...
  ],
  "warnings": ["status correlation low","drift >10%"]
}
```

### 4. `kbdiag advisor [sub] [--fix]` — 综合建议

**目标**：一条命令输出所有优化建议；`--fix` 额外输出可直接粘贴执行的 SQL。

#### 子命令

| 子命令 | 分析内容 | --fix 生成 SQL |
|--------|----------|----------------|
| `index` | 未使用/重复/缺失索引 | `DROP INDEX ...` / `CREATE INDEX ...` |
| `vacuum` | dead_pct > 20% 或 freeze 风险（xid_age > 5亿）的表 | `VACUUM (ANALYZE) schema.table;` |
| `params` | 关键参数偏离建议值 | `ALTER SYSTEM SET param = value;` + `SELECT sys_reload_conf();` |
| `analyze` | 统计信息过期表（drift > 10% 或超 7 天未 analyze） | `ANALYZE schema.table;` |

默认（无子命令）：运行全部四项。

#### 参数建议规则（`advisor params`）

| 参数 | 建议逻辑 |
|------|----------|
| `shared_buffers` | 建议为系统总内存 25%（若当前 < 20% 则警告） |
| `work_mem` | 建议 = total_mem / max_connections / 4（若当前过低则警告） |
| `checkpoint_completion_target` | 建议 0.9（若 < 0.7 则警告） |
| `wal_buffers` | 建议 64MB（若 = -1 即自动则跳过） |
| `effective_cache_size` | 建议 = 系统内存 × 75%（若差距 > 25% 则警告） |

系统内存通过 `/proc/meminfo` 获取（Linux），macOS 回退跳过内存相关建议。

#### 输出（text 模式，默认 all）

```
==> Advisor: Index
[WARN]  3 unused indexes (total 24 MB) — run 'idx unused' for detail
[WARN]  1 duplicate index pair on public.order_items
[OK]    No missing FK indexes found

==> Advisor: Vacuum
[WARN]  public.events: dead_pct=34%, 280,000 dead tuples — needs VACUUM
[OK]    No freeze risk tables (all xid_age < 500M)

==> Advisor: Params
[WARN]  shared_buffers=128MB — recommend 2048MB (25% of 8GB RAM)
[OK]    checkpoint_completion_target=0.9 ✓

==> Advisor: Analyze
[WARN]  public.orders: drift=10.4%, last analyze 3 days ago
[WARN]  public.events: last analyze 12 days ago
```

#### `--fix` 附加输出

```
-- Fix: Index
DROP INDEX CONCURRENTLY public.idx_orders_status;
DROP INDEX CONCURRENTLY public.idx_oi_notes_dup;

-- Fix: Vacuum
VACUUM (ANALYZE) public.events;

-- Fix: Params
ALTER SYSTEM SET shared_buffers = '2048MB';
SELECT sys_reload_conf();

-- Fix: Analyze
ANALYZE public.orders;
ANALYZE public.events;
```

> `--fix` 只输出 SQL，不执行。DBA 审查后手动执行。

## 测试要求

每个测试文件覆盖：

1. **happy path**：有数据时输出正确
2. **empty path**：无问题时输出 `[OK]` 不报错
3. **exit code**：有 WARN 时 exit 1，有 FAIL 时 exit 2，全 OK 时 exit 0
4. **--dry-run**（kill 命令）：不执行实际操作
5. **--format json**：`python3 -m json.tool` 验证合法 JSON
6. **--fix**（advisor 命令）：输出包含合法 SQL 关键字
7. **两节点差异**：node1（primary）/ node2（standby）分别验证
